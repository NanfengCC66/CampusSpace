**版本**：v1.1  
**编写日期**：2026-07-17

## 1. 引言

### 1.1 编写目的

本文档旨在明确CampusSpace平台的功能需求、非功能需求和约束条件，为系统设计、开发、测试和验收提供依据。

目标读者：产品经理、架构师、开发工程师、测试工程师、项目管理人员。

### 1.2 项目背景

高校教室和会议室资源管理面临以下问题：
- 资源利用率低：部分场地闲置，部分场地过度拥挤
- 预约冲突频发：缺乏有效的冲突检测和协调机制
- 人工管理效率低：传统纸质或Excel管理方式耗时耗力
- 信息不透明：用户难以快速找到合适的场地

CampusSpace平台旨在通过信息化手段解决上述问题，提升场地资源管理效率和用户体验。

### 1.3 术语定义

| 术语 | 定义 |
|-----|------|
| 场地 | 指教室、会议室、实验室等可预约使用的物理空间 |
| 固定课表 | 按学期排定的常规教学使用计划，具有较高优先级 |
| 临时预约 | 除固定课表外的临时性场地使用申请 |
| 维修停用 | 场地因维护、维修等原因暂时不可用的状态 |
| 冲突检测 | 判断新预约与已有预约、固定课表、维修时间是否重叠的过程 |
| 推荐算法 | 根据用户需求自动匹配最合适场地的算法 |

## 2. 项目概述

### 2.1 项目目标

**核心目标**：
1. 实现场地资源的数字化、可视化管理
2. 提供智能推荐功能，帮助用户快速找到合适场地
3. 建立完善的冲突检测机制，避免预约冲突
4. 支持固定课表、临时预约、维修停用等多种场景
5. 提供审批流程，确保场地使用合规
6. 统计分析场地利用率，为资源优化提供数据支持

**量化指标**：
- 预约冲突率降低至5%以下
- 用户查找场地时间缩短50%以上
- 场地利用率提升至70%以上
- 管理员工作效率提升60%以上

### 2.2 用户角色

#### 2.2.1 普通用户

**角色描述**：在校学生和教师

**主要职责**：
- 查询场地信息和可用状态
- 提交场地预约申请
- 查看个人预约记录
- 取消未开始的预约
- 查看预约审批状态
- 修改待审批或已审批的预约

**权限范围**：
- 查看所有场地基本信息（所有场地对所有用户可见）
- 提交临时预约申请（需管理员审批）
- 管理个人预约记录（不能查看他人预约详情）
- 查看个人使用统计
- 修改预约（修改后重新进入审批流程）

**角色优先级**：教师预约优先级 > 学生预约优先级

#### 2.2.2 管理员

**角色描述**：教务管理人员、场地管理人员、维修人员

**主要职责**：
- 管理场地基础信息
- 维护固定课表（支持教务系统自动导入）
- 设置维修停用时间
- 审批临时预约申请
- 审批维修申请
- 查看场地使用统计
- 处理异常情况
- 导出统计数据（需脱敏）

**权限范围**：
- 场地信息的增删改查
- 固定课表的导入和管理（实时更新）
- 维修停用的设置和取消
- 预约申请的审批和拒绝
- 维修申请的审批
- 统计数据的查看和导出（Excel/PDF格式，需脱敏）
- 强制取消已有预约
- 管理员提交的预约无需审批

**权限分配**：管理员按场地/院系分配审批权限（不分级，所有管理员权限相同）
- 统计数据的查看和导出
- 强制取消已有预约

### 2.3 业务流程

#### 2.3.1 预约申请流程

```
用户填写需求 → 系统推荐场地 → 用户选择场地 → 提交预约申请 
→ 冲突检测 → 待审批 → 管理员审批 → 审批通过/拒绝 → 使用场地
```

#### 2.3.2 场地管理流程

```
管理员添加场地 → 设置场地属性 → 录入固定课表 → 场地可用 
→ 用户可预约 → 定期统计利用率 → 优化资源配置
```

#### 2.3.3 维修管理流程

```
发现场地问题 → 提交维修申请 → 设置停用时间 → 通知受影响用户 
→ 维修完成 → 恢复场地可用
```

## 3. 功能需求

### 3.1 普通用户功能

#### 3.1.1 场地查询

**功能描述**：用户可按条件查询场地信息

**输入条件**：
- 场地名称（模糊搜索）
- 场地类型（教室/会议室/实验室）
- 容纳人数范围
- 设备需求（投影仪、空调、电脑等）
- 楼宇/校区
- 可用时间段

**输出结果**：
- 符合条件的场地列表
- 每个场地的详细信息（位置、容量、设备、图片等）
- 场地在指定时间段的可用状态

**业务规则**：
- 支持多条件组合查询
- 支持按相关度、容量、距离等排序
- 分页显示查询结果

#### 3.1.2 智能推荐

**功能描述**：系统根据用户需求自动推荐最合适的场地

**输入条件**：
- 使用时间（开始时间、结束时间）
- 参与人数
- 设备需求（可多选）
- 其他偏好（如：距离、楼宇等）

**输出结果**：
- 推荐场地列表（按匹配度排序）
- 每个场地的匹配度评分
- 推荐理由说明

**业务规则**：
- 优先推荐完全匹配的场地
- 如无完全匹配，推荐相近场地并说明差异
- 排除已被预约、维修停用、有固定课表的场地
- 支持查看推荐场地的详细信息和日历视图

#### 3.1.3 提交预约

**功能描述**：用户选择场地后提交预约申请

**输入信息**：
- 场地ID
- 使用时间
- 参与人数
- 使用目的/活动名称
- 联系方式
- 备注（可选）

**处理流程**：
1. 系统进行冲突检测
2. 如有冲突，提示冲突信息并返回
3. 如无冲突，创建预约记录（状态：待审批）
4. 发送通知给相关管理员

**业务规则**：
- 预约时间必须在当前时间之后
- 预约时长需符合系统限制（最短30分钟，最长8小时）
- 同一用户每天最多预约3次或总时长不超过12小时
- 预约需区分用途类型（教学/会议/活动/自习）
- 教师预约优先级高于学生预约
- 所有普通用户预约都需要管理员审批
- 管理员自己提交的预约无需审批

#### 3.1.4 查看预约

**功能描述**：用户查看自己的预约记录

**查询条件**：
- 预约状态（全部/待审批/已通过/已拒绝/已取消）
- 时间范围
- 场地名称

**显示信息**：
- 预约编号
- 场地名称和位置
- 预约时间
- 预约状态
- 审批意见（如有）
- 操作按钮（查看详情、取消等）

#### 3.1.5 取消预约

**功能描述**：用户取消未开始的预约

**前置条件**：
- 预约状态为"待审批"或"已通过"
- 预约开始时间未到
- 距离开始时间至少2小时
- 已开始的预约不能取消

**处理流程**：
1. 用户点击取消按钮
2. 系统确认取消原因（可选）
3. 更新预约状态为"已取消"
4. 释放场地时间资源
5. 发送通知给审批人（站内消息+短信）

**角色权限**：
- 学生：可取消自己的预约
- 教师：可取消自己的预约
- 管理员：可强制取消任何预约

#### 3.1.6 修改预约

**功能描述**：用户修改已提交或已审批的预约

**前置条件**：
- 预约状态为"待审批"或"已通过"
- 预约开始时间未到

**可修改字段**：
- 使用时间
- 参与人数
- 使用目的
- 联系方式
- 备注

**处理流程**：
1. 用户修改预约信息
2. 系统进行冲突检测
3. 如有冲突，提示并阻止修改
4. 如无冲突，更新预约信息
5. 状态重置为"待审批"
6. 发送通知给管理员

**业务规则**：
- 待审批状态的预约修改后重新进入待审批
- 已审批的预约修改后需要重新审批

### 3.2 管理员功能

#### 3.2.1 场地管理

**功能描述**：管理员维护场地基础信息

**场地属性**：
- 场地名称
- 场地类型（教室/会议室/实验室/活动室）
- 所属楼宇和楼层
- 容纳人数
- 面积
- 设备清单（投影仪、空调、电脑、白板、音响等）
- 场地图片
- 负责人/联系方式
- 备注

**操作功能**：
- 新增场地
- 编辑场地信息
- 删除场地（需检查是否有关联数据）
- 批量导入场地（Excel）
- 查看场地详情

#### 3.2.2 固定课表管理

**功能描述**：管理员维护固定课表

**课表信息**：
- 场地
- 课程名称
- 授课教师
- 上课时间（周几、第几节、开始周、结束周）
- 班级/人数

**操作功能**：
- 手动添加课表
- 从教务系统自动导入课表（实时更新）
- 批量导入课表（Excel）
- 编辑课表
- 删除课表
- 查看课表日历视图

**业务规则**：
- 课表导入前需进行冲突检测
- 课表时间段的场地不可被临时预约
- 支持按学期管理课表
- 如导入的课表存在冲突，系统提示学生找教务解决课程冲突问题
- 临时调课情况，按照真实课表在系统实时更新
- 优先级：维修停用 > 固定课表 > 临时预约

#### 3.2.3 维修停用管理

**功能描述**：管理员和维修人员设置场地维修停用时间

**停用信息**：
- 场地
- 停用原因
- 停用开始时间
- 停用结束时间（预计）
- 备注

**操作功能**：
- 维修人员提交维修申请
- 管理员审批维修申请
- 新增维修停用
- 编辑停用信息
- 维修完成后，维修人员提交恢复场地可用状态申请
- 管理员审批恢复申请
- 查看停用历史

**业务规则**：
- 维修申请需要审批
- 停用期间的场地不可预约
- 维修期间已有的预约自动取消并通知用户（提前24小时通知）
- 停用结束后自动恢复场地可用
- 优先级：维修停用 > 固定课表 > 临时预约

#### 3.2.4 预约审批

**功能描述**：管理员审批用户的预约申请

**待审批列表**：
- 显示所有待审批的预约申请
- 支持按时间、场地、申请人筛选
- 显示预约详情（申请人、时间、目的、人数等）

**审批操作**：
- 通过：更新状态为"已通过"，发送通知给申请人（站内消息+短信）
- 拒绝：更新状态为"已拒绝"，填写拒绝理由，发送通知（站内消息+短信）
- 批量审批：支持批量通过或拒绝

**业务规则**：
- 审批时需再次进行冲突检测（防止并发冲突）
- 记录审批人和审批时间
- 支持审批历史查询
- 审批时限：24小时内完成审批
- 超时未审批，系统提醒管理员
- 管理员按场地/院系分配审批权限
- 不需要多级审批流程

#### 3.2.5 统计分析

**功能描述**：管理员查看场地使用统计数据

**统计维度**：
- 按场地统计：各场地的预约次数、使用时长、利用率
- 按时间统计：日/周/月的预约趋势（默认周期：周）
- 按类型统计：不同场地类型的使用情况
- 按院系/楼宇统计：各院系/楼宇的预约情况
- 按时间段统计：工作日/周末、白天/晚上的使用情况
- 按预约类型统计：教学/会议/活动/自习的使用情况

**展示方式**：
- 数据表格
- 图表展示（柱状图、折线图、饼图）
- 支持导出Excel/PDF格式（需数据脱敏）
- 支持自定义时间范围查询

**关键指标**：
- 场地利用率 = 实际使用时长 / 场地总可用时间
- 分子包括：固定课表占用时间、已通过预约时间
- 分子不包括：维修停用时间、已取消的预约
- 分母：场地总可用时间（考虑维修停用时间）
- 预约成功率 = 审批通过数 / 总申请数
- 冲突率 = 冲突次数 / 总申请数

#### 3.2.6 强制取消预约

**功能描述**：管理员强制取消已有预约

**使用场景**：
- 场地临时维修需要取消预约
- 用户违规使用需要取消后续预约
- 其他特殊情况

**处理流程**：
1. 管理员选择要取消的预约
2. 填写取消原因
3. 系统更新预约状态为"已取消（管理员）"
4. 发送通知给预约人（站内消息+短信）

#### 3.2.7 设备状态管理

**功能描述**：管理员和维修人员管理设备状态

**设备状态**：
- 正常
- 维修中
- 已损坏

**操作功能**：
- 查看设备状态
- 更新设备状态
- 设备使用记录查询

**业务规则**：
- 设备固定在场地，不可移动
- 设备故障时，影响场地推荐（降低推荐评分）
- 需要设备使用记录

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 要求 |
|-----|------|
| 响应时间 | 页面加载时间 < 3秒，查询响应时间 < 1秒 |
| 并发用户 | 支持100个并发用户同时操作 |
| 数据量 | 支持1000个场地，10000条预约记录 |
| 推荐算法 | 推荐计算时间 < 2秒 |

### 4.2 安全需求

- 用户认证：采用Django内置认证系统，支持用户名密码登录
- 权限控制：基于角色的访问控制（RBAC）
- 数据安全：敏感信息加密存储，防止SQL注入
- 操作日志：记录关键操作日志，支持审计
- 会话管理：会话超时自动登出，防止会话劫持

### 4.3 可用性需求

- 界面友好：采用Bootstrap响应式设计，支持PC和移动端
- 操作简便：核心功能操作步骤不超过3步
- 错误提示：提供清晰的错误提示和操作引导
- 帮助文档：提供用户使用手册和FAQ

### 4.4 可维护性需求

- 代码规范：遵循PEP 8编码规范
- 文档完善：提供设计文档、API文档、部署文档
- 日志记录：完善的日志系统，便于问题排查
- 配置管理：关键参数可配置，无需修改代码

### 4.5 可扩展性需求

- 数据库切换：支持SQLite/MySQL/PostgreSQL切换
- 功能扩展：模块化设计，便于功能扩展
- 接口预留：预留API接口，支持与其他系统集成

- 部署环境：支持Linux服务器部署
- 浏览器兼容：支持Chrome、Firefox、Edge最新版本

## 6. 数据需求

### 6.1 数据实体

主要数据实体包括：
- 用户（User）
- 场地（Venue）
- 预约（Booking）
- 固定课表（Schedule）
- 维修停用（Maintenance）
- 设备（Equipment）
- 审批记录（Approval）
- 操作日志（Log）

### 6.2 数据关系

- 一个场地可以有多个预约记录
- 一个场地可以有多条固定课表
- 一个场地可以有多条维修记录
- 一个预约对应一个审批记录
- 一个用户可以有多个预约记录

### 6.3 数据量估算

| 数据类型 | 年增长量 | 保留年限 | 总量估算 |
|---------|---------|---------|---------|
| 场地 | 50 | 5年 | 250条 |
| 预约记录 | 10000 | 2年 | 20000条 |
| 固定课表 | 5000 | 5年 | 25000条 |
| 用户 | 5000 | 5年 | 25000人 |

## 7. 接口需求

### 7.1 用户界面

- 响应式Web界面，支持PC和移动设备
- 提供日历视图展示场地占用情况

## 9. 业务规则总结

### 9.1 审批流程规则

| 规则项 | 规则内容 |
|-------|---------|
| 审批范围 | 所有普通用户预约都需要管理员审批 |
| 管理员预约 | 管理员自己提交的预约无需审批 |
| 审批时限 | 24小时内完成审批，超时提醒管理员 |
| 审批权限 | 管理员按场地/院系分配审批权限 |
| 审批流程 | 单级审批，无需多级审批 |

### 9.2 冲突检测规则

| 规则项 | 规则内容 |
|-------|---------|
| 优先级 | 维修停用 > 固定课表 > 临时预约 |
| 场地独占 | 一个时间段内，一个场地只能有一个预约 |
| 分区使用 | 不支持场地分区使用 |
| 时间间隔 | 前后预约需要10分钟间隔 |

### 9.3 推荐算法规则

| 规则项 | 规则内容 |
|-------|---------|
| 权重分配 | 容量匹配度、设备匹配度、类型匹配度、历史偏好 |
| 利用率平衡 | 不考虑场地利用率平衡 |
| 模糊匹配 | 支持容量稍大、缺少部分设备的场地推荐 |
| 结果数量 | 返回所有符合条件的，支持分页 |

### 9.4 取消修改规则

| 规则项 | 规则内容 |
|-------|---------|
| 取消时限 | 开始前2小时内不能取消 |
| 已开始预约 | 已开始的预约不能取消 |
| 取消通知 | 通知相关人员（站内消息+短信） |
| 修改规则 | 待审批和已审批的预约都能修改，修改后重新审批 |
| 可修改字段 | 时间、人数、设备、用途等所有字段 |

### 9.5 利用率统计规则

| 规则项 | 规则内容 |
|-------|---------|
| 统计周期 | 默认周期：周，支持自定义时间范围 |
| 分子计算 | 包括固定课表和已通过预约，不包括维修停用和已取消预约 |
| 分母计算 | 场地总可用时间（考虑维修停用时间） |
| 统计维度 | 场地、院系/楼宇、时间段、预约类型 |

### 9.6 权限管理规则

| 规则项 | 规则内容 |
|-------|---------|
| 管理员分级 | 不分级，所有管理员权限相同 |
| 场地可见性 | 所有场地对所有用户可见 |
| 预约可见性 | 用户不能查看他人预约详情 |
| 数据导出 | 所有管理员可导出，需数据脱敏 |

### 9.7 设备管理规则

| 规则项 | 规则内容 |
|-------|---------|
| 设备属性 | 设备固定在场地，不可移动 |
| 设备状态 | 正常/维修中/已损坏 |
| 故障影响 | 设备故障影响场地推荐 |
| 使用记录 | 需要设备使用记录 |

### 9.8 固定课表规则

| 规则项 | 规则内容 |
|-------|---------|
| 导入方式 | 支持教务系统自动导入 |
| 更新频率 | 实时更新 |
| 冲突处理 | 系统提示学生找教务解决课程冲突问题 |
| 临时调课 | 按照真实课表在系统实时更新 |

### 9.9 维修停用规则

| 规则项 | 规则内容 |
|-------|---------|
| 申请流程 | 维修人员提交申请，需要审批 |
| 恢复流程 | 维修完成后提交恢复申请，需要审批 |
| 影响处理 | 维修期间已有预约自动取消并通知用户 |
| 提前通知 | 提前24小时通知用户 |

### 9.10 其他业务规则

| 规则项 | 规则内容 |
|-------|---------|
| 预约时长 | 最短30分钟，最长8小时 |
| 预约限制 | 同一用户每天最多预约3次或总时长不超过12小时 |
| 用途分类 | 教学/会议/活动/自习 |
| 角色优先级 | 教师预约优先级 > 学生预约优先级 |
| 黑名单机制 | 多次违规限制预约权限（具体标准待定） |
